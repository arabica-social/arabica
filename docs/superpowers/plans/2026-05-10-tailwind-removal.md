# Tailwind Removal Implementation Plan

**Goal:** Drop Tailwind CSS as a build dependency while preserving the current visual and behavior exactly. Keep utility class names in markup unchanged; replace the engine that generates the CSS for them.

**Architecture:** Three-file plain-CSS pipeline served as a single concatenated `output.css`. No build step besides concatenation + optional minification. Class names in `.templ` files and JS stay byte-identical. Dark mode keeps its existing two-track strategy (`prefers-color-scheme` media query + `[data-theme="dark"]` attribute).

**Tech Stack:** Plain CSS3, CSS custom properties, `@media` for breakpoints + theme, `:hover` / `:focus` / sibling selectors for state. No `@apply`, no `@source`, no Tailwind config. `cat` (or equivalent) is the only build tool; `lightningcss` optional for minification.

**Scope clarification:** This plan removes Tailwind, *not* the utility-class-first authoring style. Templates still say `class="flex items-center gap-3"`; those classes are now defined in our `utilities.css` instead of generated by Tailwind. Migrating templates to semantic component classes is a *separate* future project.

## Type System

These types are the contract every phase implements against. Defined upfront so later tasks don't drift.

### Token types

Three categories. Each token has a fixed category and rules about when it may vary.

**`PaletteToken`** — frozen across themes. Raw brand colors.
```
Shape: --{brand}-{name}-{shade}  (shade ∈ 50,100,…,900,950)
Examples:
  --brand-brown-50 … --brand-brown-950
  --brand-amber-50 … --brand-amber-800
  --brand-red-50, --brand-red-100, --brand-red-200, --brand-red-300,
    --brand-red-500, --brand-red-700, --brand-red-800, --brand-red-900
  --brand-green-{100,200,500,600,700,800}
  --brand-blue-{100,200,400,700,800}
  --brand-purple-{100,200,700,800}
  --brand-gray-{100,800}
Constraint: Identical value in :root and in dark-mode blocks.
```

**`SemanticToken`** — theme-aware. Aliases to palette tokens, but the alias swaps in dark mode.
```
Shape: --{role}-{slot}
Examples:
  Text:        --text-primary, --text-secondary, --text-muted, --text-faint, --text-placeholder
  Surface:     --card-bg, --surface-bg, --journal-bg, --kraft-bg, --feed-card-bg, --feed-board-bg
  Border:      --card-border, --card-border-strong, --feed-board-border
  Accent:      --btn-primary-bg, --btn-primary-bg-hover, --btn-primary-text,
               --input-border-focus
  Feedback:    --rating-bg, --rating-text
  Shadow:      --card-shadow, --card-shadow-hover, --shadow-sm, --shadow-md, --shadow-lg
Constraint: MUST resolve to a PaletteToken or a literal hex inside the same :root or [data-theme] block.
```

**`TypeTagToken`** — entity-type accent colors (brew/bean/recipe/roaster/grinder/brewer).
```
Shape: --type-{entity}
Examples: --type-brew, --type-bean, --type-recipe, --type-roaster, --type-grinder, --type-brewer
Constraint: Single value per theme; used by feed cards and filter pills.
```

### Utility class types

A `UtilityClass` is `(name, css-body)` where `name` matches the literal token in `.templ` files. Variants compose around it.

**`SimpleUtility`** — no token lookup, fixed CSS.
```
Examples:
  .flex { display: flex; }
  .hidden { display: none; }
  .truncate { overflow: hidden; text-overflow: ellipsis; white-space: nowrap; }
```

**`ScaleUtility`** — value derived from a numeric scale. `n` maps to `n * 0.25rem` (the Tailwind spacing scale we already use).
```
Shape: .{family}-{n}
Families: p, px, py, pt, pb, pl, pr, m, mx, my, mt, mb, ml, mr,
          gap, gap-x, gap-y, space-x, space-y,
          w, h, min-w, max-w, min-h, max-h,
          top, bottom, left, right, inset
Scale values used (audit-confirmed): 0, 0.5, 1, 1.5, 2, 2.5, 3, 3.5, 4, 5, 6, 7, 8, 10, 12, 14, 16, 20, 24
Special suffixes: -auto, -full, -screen, -px, -min, -max, -fit
Example: .px-4 { padding-left: 1rem; padding-right: 1rem; }
```

**`ColorUtility`** — references a PaletteToken or a literal.
```
Shape: .{prop}-{color-name}-{shade}[/{opacity}]
Props: text, bg, border, ring, decoration, divide, placeholder
Body:
  .text-brown-700 { color: var(--brand-brown-700); }
  .bg-brown-50/60 { background-color: color-mix(in srgb, var(--brand-brown-50) 60%, transparent); }
  .text-white { color: #FFFFFF; }
  .bg-transparent { background-color: transparent; }
Opacity modifier /{N} → color-mix() with transparent at (100-N)%.
```

**`TypographyUtility`** — references typography tokens.
```
.text-xs, .text-sm, .text-base, .text-lg, .text-xl, .text-2xl, .text-3xl
.font-normal, .font-medium, .font-semibold, .font-bold
.tracking-{tighter,tight,normal,wide,wider,widest}
.leading-{none,tight,snug,normal,relaxed,loose}
.italic, .uppercase, .lowercase, .capitalize, .underline, .line-through, .no-underline
```

**`EffectUtility`** — shadow, opacity, transitions.
```
.shadow-{sm,md,lg,xl,2xl,inner,none}
.opacity-{0,25,50,60,75,80,90,100}
.transition, .transition-{colors,opacity,transform,all}
.duration-{75,100,150,200,300,500,700,1000}
.ease-{linear,in,out,in-out}
```

**`LayoutUtility`** — flex/grid/position primitives.
```
.flex, .inline-flex, .grid, .block, .inline-block, .inline, .hidden, .table, .contents
.flex-{row,col,row-reverse,col-reverse}, .flex-{1,auto,initial,none,wrap,nowrap,wrap-reverse}
.items-{start,end,center,baseline,stretch}
.justify-{start,end,center,between,around,evenly}
.self-{start,end,center,stretch,auto}
.grid-cols-{1,2,3,4,5,6,12}, .col-span-{N}, .row-span-{N}
.relative, .absolute, .fixed, .sticky, .static
.z-{0,10,20,30,40,50,auto}
```

**`BorderUtility`** — border width + radius.
```
.border, .border-{0,2,4,8}, .border-{t,r,b,l,x,y}, .border-{t,r,b,l}-{2,4}
.rounded, .rounded-{none,sm,md,lg,xl,2xl,3xl,full}, .rounded-{t,r,b,l,tl,tr,bl,br}-{sm,md,lg,…}
.ring, .ring-{0,1,2,4,8}, .ring-inset, .ring-offset-{0,1,2,4}
```

### Variant types

A `Variant` wraps a UtilityClass in additional selector/at-rule logic.

**`StateVariant`** — pseudo-class wrappers.
```
Shape: .{variant}\:{utility-name}{selector-suffix}
Generators:
  hover:X    →  .hover\:X:hover { …X-body… }
  focus:X    →  .focus\:X:focus { …X-body… }
  disabled:X →  .disabled\:X:disabled { …X-body… }
  active:X   →  .active\:X:active { …X-body… }
Confirmed in use: hover (135), focus (31), disabled (a few).
```

**`GroupVariant`** — parent-scoped state.
```
Shape: .group:hover .group-hover\:X  or  .group\/{name}:hover .group-hover\/{name}\:X
Confirmed in use: group-hover/author (6 sites; named group).
Must generate both unnamed and named forms for every group-hover utility actually used.
```

**`ResponsiveVariant`** — min-width media wrappers.
```
sm: → @media (min-width: 640px)
md: → @media (min-width: 768px)
lg: → @media (min-width: 1024px)
Confirmed in use: sm (50), md (10), lg (18). xl and 2xl unused.
```

**`ThemeVariant`** — `dark:` prefix. **NOT used in templates** (audit confirms 0 hits). Dark mode runs through the override block in `app.css:270-330` that targets specific class names. We preserve that mechanism; no `dark:` generation needed.

### File system contract

```
static/css/
├── tokens.css        ← all PaletteToken + SemanticToken + TypeTagToken definitions
├── reset.css         ← preflight equivalent (~30 rules)
├── utilities.css     ← every UtilityClass × Variant in use, hand-written or generated
├── components.css    ← bespoke component classes (existing app.css content minus @apply)
├── themes/oolong.css ← unchanged
├── output.css        ← concatenation of (tokens + reset + utilities + components), served to browser
└── app.css           ← DELETED at end of plan
```

The served file is the concatenation. The split exists for human authoring.

### Build pipeline contract

```
Input:  tokens.css, reset.css, utilities.css, components.css
Output: output.css (concatenated, optionally minified)
Command: cat <inputs> > output.css  [|  lightningcss --minify -i - -o output.css ]
Recipe: justfile `style-arabica`
No watch mode needed; CSS files change rarely enough that re-running `just style` is acceptable.
Cache buster: bump v= query in internal/web/components/layout.templ after every output.css change.
```

### Dark mode contract

The existing override mechanism in `app.css:270-330` is:
- `@media (prefers-color-scheme: dark) { .text-brown-900 { color: var(--text-primary) !important; } … }`
- Mirrored `:root[data-theme="dark"] .text-brown-900 { … }` block

This block depends on the literal class name `.text-brown-900` (and friends) existing on elements. As long as `utilities.css` generates `.text-brown-900 { color: var(--brand-brown-900); }`, the override has a target.

**Rule:** every theme-overridden utility MUST also have a base utility in `utilities.css`. Phase 2 enforces this with a check.

## Risk Register

Captured upfront so a future engineer can sanity-check decisions:

**Risk 1: Missed utility class.** A class used in a `templ.KV("rare-utility-string", cond)` call doesn't appear in the inventory because grep's class-extraction is approximate. Mitigation: Task 2.1 also scans `templ.KV(...)` and `templ.Classes(...)` calls. Acceptance: open every page, watch DevTools for missing-rule warnings.

**Risk 2: HTMX-swapped content references a class that worked at first load but not after.** HTMX returns fragments that may inject classes; those classes still need rules. Mitigation: the inventory scan covers full `.templ` files including HTMX response templates. All templ files are scanned, not just pages.

**Risk 3: Dark mode override depends on `!important` ordering that changes when we move it.** The current overrides use `!important`. As long as they remain in cascade position after the base utilities, they win. Mitigation: keep the override block in `components.css` and concat AFTER `utilities.css`.

**Risk 4: Tailwind's Preflight does things we depend on without realizing.** Examples: `button` default styling reset, `img` block display. Mitigation: Task 2.2 ports Preflight to `reset.css`; visual sweep in Phase 5 catches anything missed.

**Risk 5: Arbitrary-value sites missed.** Mitigation: explicit inventory in Task 2.7.

**Risk 6: Theme switcher (`[data-theme="dark"]`) and media query (`prefers-color-scheme`) both fire and conflict.** They already coexist today — no change. Mitigation: leave the dual mechanism intact.

**Risk 7: Oolong theme breaks.** The oolong sister app shares the same engine. Mitigation: Task 4.1 updates oolong's recipe; Task 5.1 explicitly tests the oolong build.
